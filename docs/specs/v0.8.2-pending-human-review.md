# Harmonia — design spec extension v0.8.2: the `pending_human_review` provenance state

**Status:** **implemented in v0.8.2** (2026-06-23) · extends [`spec.md`](../../spec.md) (v0.1)
and [v0.8](v0.8-machine-crosscheck.md) · hardens the **provenance/trust surface**
(§9) · **adds no modeling and changes no risk number. It does NOT promote any
record to `verified`.**

> *Until now every record carried a binary honesty flag: `unverified` (the default
> for all 104 records) or `verified` (which an LLM may never set — spec §9). That
> conflated two very different things: a value transcribed from a real, citable
> source that an independent published source corroborates, versus an illustrative
> placeholder nobody has checked. v0.8.2 adds the honest middle state —
> `pending_human_review` — and fills it by fetching real published data and
> corroborating Harmonia's values against it. A record reaches this state ONLY by
> machine corroboration against an independent published number; it still requires
> a human reading the source PDF to become `verified`.*

---

## 0. Why this, and why now

The single biggest gap between Harmonia-as-demonstrator and Harmonia-as-resource
is the `0/104 verified` count (§9). Closing it fully is human curatorial labor.
But most of the *risk* in an unverified value — that it is fabricated, mis-keyed,
or unit-slipped — is mechanically reducible by checking it against an independent
published number, which the v0.8 cross-check already does for the 12 training
drugs. v0.8.2 turns that check into a first-class provenance state and extends its
coverage to the validation drugs by sourcing their published potencies.

This pays down verification debt without ever claiming a human did the reading.

## 1. The state

`extraction.review_status` gains `pending_human_review` (schema enum, between
`unverified` and `verified`). Semantics:

| State | Meaning |
| --- | --- |
| `unverified` | Literature-derived but **uncorroborated** — no independent published source has been matched, or the matched source disagrees. Treat as illustrative. |
| `pending_human_review` | The IC50 **agrees (≤5×) with an independent published value** (provenance recorded in `extraction.corroboration`). Sourced and machine-corroborated — but **no human has read the source PDF**. |
| `verified` | A human contributor confirmed the value against the primary source and added themselves to `extraction.verified_by`. **An LLM/automated process may never set this** (§9). |

A `pending_human_review` record carries an `extraction.corroboration` block: the
corroborating `source`, its `citation` key, and the `ic50_fold_diff`. The state is
**computed deterministically** by the build tool from the reference tables, so it
is reproducible (CI gate) and never hand-set.

## 2. Eligibility (how a record is promoted)

`build_records.corroborate()` promotes a channel-block record from `unverified`
to `pending_human_review` iff **all** hold:

1. The channel is **identifiable** (`max_block ≥ 60`). A Tier-D channel's IC50 is
   an extrapolation, so "sourced and awaiting confirmation" would mislead — those
   stay `unverified`.
2. An independent published reference value exists for that drug × channel.
3. The record's IC50 agrees with it within **5×** — the dataset's own divergence
   threshold (a several-fold cross-lab/method gap is expected, not an error; a
   larger gap means the value is *not* corroborated, so it stays `unverified` and
   honestly flags a discrepancy a human should resolve).

Everything else — uncorroborated channels, all non-channel-block records
(drug-reference, AP-model, population) — stays `unverified`. Coverage is reported,
never assumed.

## 3. The reference data (what was sourced, and how it was verified)

Two published reference tables under `dataset/references/`, each built
deterministically and credited per source:

- **Training drugs (12) — `cipa_block_reference.json`** (from v0.8): Li et al.
  2017 multichannel IC50/Hill, via the FDA/CiPA `Li2017_IC50.csv`. Also drives the
  `harmonia crosscheck` transcription-error detector.
- **Validation drugs (16) — `cipa_validation_reference.json`** (new):
  - **hERG/IKr — Ridder et al. 2020 (`ridder-2020`, new citation), Table 1**: the
    Bayesian-hierarchical static hERG IC50. **Re-verified value-by-value against
    the published PMC table (PMC7166077) — all 16 transcribe exactly.**
  - **ICaL — Li et al. 2019 (`li-2019`), Supplement S2 Table 4** (µM → nM ×1000):
    **independently cross-corroborated** — these agree within ~5× with Harmonia's
    pre-existing, separately-sourced crumb-2016 ICaL values for the drugs that
    carry one (two independent transcriptions agreeing).
  - **INaL / INa deliberately excluded** (resolved, not pending): the Li-2019 fits
    for those currents are very poorly identified — credible intervals spanning
    several orders of magnitude (e.g. risperidone INaL 12,500–7.18×10⁹ nM) — so they
    are unsound corroboration anchors. Encoding them, even flagged, would assert a
    "published value" the source itself does not identify. This is a deliberate
    scope boundary (the same posture as the v0.8.4 max-block resolution), not
    unfinished work: those records keep their existing values and stay `unverified`
    until a human reads the source.

The validation table is used **only** for `pending_human_review` corroboration —
deliberately NOT fed to `harmonia crosscheck`, whose tight tolerance assumes the
same-lineage Li-2017 training table. Mixing a cross-method table into the
transcription-error detector would manufacture false "errors" from legitimate
inter-method spread.

## 4. What it produced

Of 104 records, **56 are now `pending_human_review`** (up from 0) and 48 remain
`unverified`. The promoted set: the 12 training drugs' identifiable channels
(corroborated vs Li 2017) plus the validation drugs' hERG and ICaL channels whose
values agree with Ridder 2020 / Li 2019. Honest non-promotions include
`astemizole.ikr` (its 0.9 nM Kramer value sits 21× from Ridder's 19 nM — a real
cross-method discrepancy) and `loratadine` (both channels disagree >5×); those
stay `unverified` and are exactly the records a human should look at first.

`verified` remains **0/104**. Nothing here is a human reading a PDF.

## 5. The honesty contract

1. **`pending_human_review` ≠ `verified`.** Different states, different meanings,
   reported on separate lines by `harmonia info`. Only a human adding to
   `verified_by` reaches `verified` (§9).
2. **No value is invented.** Every reference number is either re-verified against
   its published table (hERG) or independently cross-corroborated (ICaL). Numbers
   that could not be verified were not encoded.
3. **Corroboration is a gate, not a rewrite.** Promotion never changes a record's
   value; it only checks agreement and records provenance. (Where the v0.8
   cross-check found an actual error — cisapride.ical, terfenadine.inal — that was
   reconciled separately in v0.8.1, against the raw primary data.)
4. **Disagreement is surfaced, not buried.** A value that fails corroboration
   stays `unverified` rather than being silently "filled," preserving the signal
   that a human should reconcile it.

## 6. Non-drift

One schema enum value, one extraction sub-field, one reference file, one citation,
and review-status bookkeeping. No risk number, threshold, classification, export
value (beyond the review-status column), or simulation output changes. The v0.8
`harmonia crosscheck` detector and its zero-divergence guarantee are untouched.
